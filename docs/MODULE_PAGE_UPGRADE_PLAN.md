# 模块页骨架升级 · 讨论稿

> 把模块页拆成「速览区（固定结构）+ 详解区（LLM 自由发挥）」，提高对异质模块的适配性，同时减少跨模块幻觉。

---

## 一、为什么要改

### 当前骨架（`build_module_page_prompt`）
1. 职责与边界
2. 内部组成（文件表）
3. 关键代码（2-5 个符号）
4. 跨模块关系
5. 可选 mermaid

### 实际产出的问题（以 `module_5` 维基文档生成 为证）

| 现象 | 根因 |
|---|---|
| 「关键代码」4 个 bullet 互不衔接，不成叙事 | 模板要求"列符号"而非"讲流程" |
| 跨模块依赖经常虚构（依赖关系与 AST 不符） | 让 LLM 写本可由 AST 计算的事实 |
| mermaid 是可选项，非平凡模块经常省略 | "若数据流清晰，可以附一个"措辞过软 |
| 数据访问层 / 编排服务 / 配置常量 共用同一模板 | 5 段对所有模块一视同仁，与异质性冲突 |
| 部分代码锚点指向不存在的符号 | 自由发挥与 AST 没对齐，正交问题，本次一并收紧 |

### 设计目标

- **保住扫描友好性** —— wiki 价值之一是横向比较，速览结构必须稳定
- **解放 LLM 的写作选择权** —— 详解部分让 LLM 决定"怎么讲"
- **把事实留给代码** —— 跨模块依赖由 AST 直接渲染，LLM 不参与
- **不增加 LLM 调用次数** —— 单模块仍是 1 次调用

---

## 二、两段式骨架

```
┌─────────────────────────────────────┐
│  速览区（固定，半计算半 LLM）         │  ← 5 秒扫完知道坐标
├─────────────────────────────────────┤
│  详解区（LLM 自由发挥 + 反退化约束）  │  ← 真正讲清楚
└─────────────────────────────────────┘
```

### 2.1 速览区（固定）

| 项 | 内容 | 生成方式 |
|---|---|---|
| 一句话定位 | 取自 `module_split` description，必要时 LLM 微调 | LLM |
| 文件清单表 | 路径（行内 code，不链接）+ 一行职责 | LLM 写"职责"列 |
| 跨模块关系 | `→ module_N (k 处调用)` / `← module_N (k 处调用)` | **纯计算**（复用 `_fmt_cross_module`） |
| 阅读建议 | **标题固定，内容自由**：LLM 决定一句话钩子 / 场景化导引 / 重点提示 / 直接留空 | LLM |

**关键决策 1：跨模块表完全由代码侧渲染**——目前是把它当 prompt 输入交给 LLM，LLM 转手写出来时丢精度甚至编。改成在 `module_page_generator` 里直接拼到最终 `content_md` 顶部，LLM 看不到这一步。一举消灭一类幻觉。

**关键决策 2：阅读建议是"固定槽位 + 自由内容"**——和详解区一个思路。槽位标题固定（保持扫描友好），内容由 LLM 按模块情况选形态：

| 适用场景 | LLM 输出形态 |
|---|---|
| 简单/小模块 | 一句话钩子（"若只读一节，看 X"） |
| 多视角模块 | 场景化导引（"想 A 看 X / 想 B 看 Y"，2-3 条） |
| 单一显眼入口 | 直接指出入口位置 |
| 太简单不需引导 | 输出空字符串，渲染时整节不显示 |

**软约束**（写进 prompt）：
- 长度 ≤ 80 字（防止变成迷你正文）
- 必须做出"取舍/重点判断"，禁说"全文都很重要"
- **禁止纯粹复述详解区 H2 标题**（不要变成迷你目录）

### 2.2 详解区（自由）

LLM 决定结构、段数、标题、顺序。只受三类约束：

#### A. 渲染契约（硬性，原本就有，不放松）
- 代码引用必须用 `[..](#code:ref_N)`，且 `ref_N` 在 `code_refs` 中声明
- 跨模块跳转必须用 `[..](#wiki:module_N)`，且只能引用 `allowed_page_ids` 里的
- 不准链接 `#wiki:file_*`、`#wiki:overview`、`#wiki:cat_*` 等

#### B. 反退化（新增）
- **禁用通用 H2**：「概述 / 介绍 / 详情 / 总结 / 说明 / 内容」等
- **禁止复述速览区事实**：不准再列一遍文件清单 / 不准再罗列依赖
- **每个 H2 以具体观察起手**，不准空话铺垫（"本模块非常重要"这类）
- **代码锚点最少数量**：`max(2, ceil(file_count / 3))`，不达标判负样本

#### C. 角度建议（非必选清单，防 LLM 退化到三段式）

> 在以下角度中**挑 2-4 个最适合本模块**的展开，不要凑数：
> - 工作流程 / 数据流
> - 核心概念与关键抽象
> - 关键代码导览（按调用顺序串）
> - 状态与生命周期
> - 配置与扩展点
> - 设计取舍 / 易踩的坑
> - 与其他角度（你认为更合适的）

## 三、最终 Markdown 形态（示意）

```markdown
## 速览

**定位**：维基文档生成模块——把代码 / README / AST 转成可读的 WikiDocument。

**文件清单**

| 路径 | 职责 |
|---|---|
| `backend/services/wiki/outliner.py` | 大纲规划，决定章节/专题骨架 |
| `backend/services/wiki/module_page_generator.py` | 模块页生成器（本文档关心的入口） |
| ... | ... |

**跨模块关系**（由 AST 计算）
- → `module_2` (8 处调用)
- → `module_6` (3 处调用)
- ← `module_9` (5 处调用)

**阅读建议**
> 若只读一节，看「工作流程」——它把从入口到落盘的完整链路串在一起，
> 是进入本模块的最佳切入点。

---

## 工作流程

（LLM 自己决定标题，自己决定要不要画 mermaid，自己决定写多长）

## 关键抽象：DocBundle 与 OutlinePlan

（LLM 自己决定要不要这一节）

...
```

速览区由代码与 LLM 协同填，详解区从 `## 工作流程` 之后整段交给 LLM。

## 四、prompt 重写要点

### 4.1 单次 LLM 调用 + JSON 字段顺序约束

整个模块页 **一次 LLM 调用** 搞定。利用 LLM 自回归生成的特性，让 `reading_guide` 字段排在 `detail_md` 之后——LLM 写到 guide 时已经"看到"自己刚写的 detail，可以基于实际内容做取舍判断，不需要事先承诺结构（避免重蹈"硬填模板"的覆辙）。

JSON 输出 schema（**字段顺序不可变**）：

```json
{
  "speed_summary": {
    "tagline": "一句话定位 (≤30 字)",
    "file_roles": {
      "<path1>": "一行职责 (≤25 字)",
      "<path2>": "..."
    }
  },
  "detail_md": "<详解区 markdown，LLM 自由发挥>",
  "code_refs": {
    "ref_1": {"symbol": "...", "file": "..."}
  },
  "reading_guide": "<≤80 字；可空字符串>"
}
```

**最终渲染顺序**（由 `module_page_generator` 拼装）：

```
[tagline]
[file_roles 渲染成表格]
[跨模块关系——代码侧计算渲染，LLM 不参与]
[reading_guide]（若非空）
---
[detail_md]
```

注意：跨模块关系**不在** JSON 里——LLM prompt 里仍喂给它做上下文参考（避免详解区写偏），但最终渲染时由代码从 AST 直接生成。

### 4.2 prompt 结构（user 部分）

```
## 模块信息
- 名称、一句话描述

## 模块完整源码
...（同今天）

## 与其他模块的交互（仅供参考，最终渲染由代码生成，你不必复述）
...（同今天的 _fmt_cross_module 文本）

## 模块包含的文件路径（file_roles 必须为这些 path 写职责，不可增删）
- path1
- path2

## 可用页面 id（仅其他模块）
...

## 任务

按下方 schema 输出 JSON，**字段顺序不可调整**：

1. speed_summary.tagline / file_roles
2. detail_md：自由 markdown
   · 你决定段数、标题、顺序
   · 角度建议（非必选，挑 2-4 个最适合的）：工作流程 / 核心概念 /
     关键代码导览 / 状态生命周期 / 配置扩展点 / 设计取舍 / 易踩的坑
   · 禁用通用 H2：概述/介绍/详情/总结/说明/内容
   · 不准复述速览区事实
   · 至少 N 个 [..](#code:ref_X) 锚点（N = max(2, ceil(file_count/3)))
   · 跨模块跳转用 [..](#wiki:module_N)，仅引用 allowed_page_ids
3. code_refs：{ref_X: {symbol or file}}
4. reading_guide：你已写完 detail_md，回看自己的产出，
   挑出最值得读的一节并说明理由（≤80 字）。
   形态自由：一句话钩子 / 场景化导引 / 重点提示。
   模块太简单可直接输出空字符串。
```

## 六、待讨论 / 待拍板

1. **详解区角度建议清单**给不给？
   - 给：避免退化为三段式
   - 不给：完全自由，赌 LLM 能找到角度
   - 草案：**给**，但措辞为"建议"非"必选"

2. **大模块要不要走两步生成**（先 LLM 出大纲再 LLM 写正文）？
   - 阈值候选：`file_count > 8` 或 `代码总长 > MODULE_CODE_BUDGET_CHARS * 0.5`
   - 代价：×2 调用次数
   - 草案：**先不做**，等单步效果不达标再上

3. **反退化的"H2 黑名单"由 prompt 约束还是后处理校验**？
   - prompt 约束：LLM 大概率会遵守，但偶尔翻车
   - 后处理校验：检测到黑名单标题直接拒绝并重试一次
   - 草案：prompt 先做，如有翻车再加后处理

4. **详解区代码锚点下限不达标怎么办**？
   - 重试一次？补默认锚点？放过？
   - 草案：**放过 + 日志告警**，不阻塞生成

5. **阅读建议节的标题**叫什么？
   - 候选：「阅读建议」/「怎么读这一页」/「重点」/「导读」
   - 草案：**「阅读建议」**——口吻克制、不指挥读者
