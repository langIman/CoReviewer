# 从文件摘要 → 文件夹摘要 → 项目摘要

## 整体流程

```
文件摘要 → 文件夹摘要 → 项目摘要
```

自底向上，逐层聚合。每一层的摘要既是上层的输入，也是用户逐层下钻时可直接使用的产物。

## 数据持久化

所有层级的摘要生成后持久化到数据库，方便后续查询和复用。存储结构：

| 字段 | 说明 |
|------|------|
| path | 文件/文件夹/项目的路径 |
| type | 层级类型：file / folder / project |
| summary | LLM 生成的摘要文本 |

每一层摘要生成后立即写入数据库，上层聚合时直接从数据库查询下层摘要。

---

## 第一层：文件摘要

### 核心思路

对每个文件进行结构感知的信息提取，生成文件级摘要。

### 提取规则

1. **提取每个函数的前5行**
2. **提取每个类的前5行**
3. **根据文件总行数设置30%截断上限**，防止大文件提取过多 token

### 摘要生成

将提取的内容喂给 LLM，使用以下 prompt：

```
以下是文件 {file_path} 的部分代码片段（包含各函数和类的前几行）。
请根据这些片段，用一段简洁的话概括这个文件的职责和作用。
只需要描述它做什么，不要复述代码内容。
如果你认为提供的代码片段不足以判断该文件的职责，请直接输出"信息不足无法推测"，不需要任何其他回答。

{extracted_code}
```

### 重传机制

当 LLM 返回"信息不足无法推测"时，上传文件完整代码重新提取并再次请求，若还是返回"信息不足无法推测"，则文件摘要记录为"该文件/LLM出错"。

---

## 第二层：文件夹摘要

### 核心思路

当一个文件夹下所有文件的摘要生成完毕后，将这些摘要聚合起来交给 LLM，生成该文件夹的摘要。嵌套文件夹严格自底向上处理。

### 处理顺序

严格自底向上，从最深层的叶子文件夹开始：

```
agents/lead.py 摘要  ┐
agents/worker.py 摘要 ┘→ agents/ 摘要 ┐
llm/llm_service.py 摘要 → llm/ 摘要    ┘→ services/ 摘要 → ...→ 项目摘要
```

每一层文件夹的摘要生成，必须等待其所有子文件夹和子文件的摘要就绪。

### 摘要生成

将该文件夹下所有文件摘要（含子文件夹摘要）带文件名拼接，喂给 LLM：

```
以下是文件夹 {folder_path} 中各文件和子文件夹的摘要：

{file_or_subfolder_name_1}: {summary_1}
{file_or_subfolder_name_2}: {summary_2}
...

请根据这些摘要，用一段话概括这个文件夹的整体职责和作用。
只需要描述它做什么，不要逐个复述各文件的摘要。
如果你认为提供的摘要信息不足以判断该文件夹的职责，请直接输出"信息不足无法推测"，不需要任何其他回答。
```

### 重传机制

当 LLM 返回"信息不足无法推测"时，将该文件夹下各文件的完整摘要连同子文件夹的摘要重新组织后再次请求，若仍返回"信息不足无法推测"，则文件夹摘要记录为"该文件夹/LLM出错"。

---

## 第三层：项目摘要

### 核心思路

当项目根目录下所有文件夹的摘要生成完毕后，将这些摘要聚合起来交给 LLM，生成项目级摘要。

### 摘要生成

将所有顶层文件夹摘要（及根目录下的文件摘要）带名称拼接，喂给 LLM：

```
以下是项目 {project_name} 中各文件夹和文件的摘要：

{folder_or_file_name_1}: {summary_1}
{folder_or_file_name_2}: {summary_2}
...

请根据这些摘要，概括这个项目的整体职责和作用。
只需要描述它做什么，不要逐个复述各文件夹的摘要。
如果你认为提供的摘要信息不足以判断该项目的职责，请直接输出"信息不足无法推测"，不需要任何其他回答。
```

### 重传机制

当 LLM 返回"信息不足无法推测"时，将所有文件夹摘要重新组织后再次请求，若仍返回"信息不足无法推测"，则项目摘要记录为"该项目/LLM出错"。

---

项目摘要是整个自底向上摘要链的最终产物，也是用户认知项目的第一入口。
