# CoReviewer 图生成逻辑

## 整体流程概览

```
用户点击"可视化" → UploadBar.handleVisualize()
                        │
          ┌─────────────┴──────────────┐
          ▼ (并行, 先到先显示)           ▼ (等待 LLM 完成后替换)
   analyzeGraph()                analyzeOverview()
   POST /api/analyze/graph       POST /api/analyze/overview
   纯 AST, 毫秒级                 AST骨架 → LLM 语义化
          │                              │
          ▼                              ▼
   模块级骨架图 (占位)            语义化流程图 (最终展示)
          │                              │
          └──────────┬───────────────────┘
                     ▼
              ResponseCard 解析 JSON
              传入 <FlowChart data={...} />
                     │
                     ▼
              Dagre 布局 → ReactFlow 渲染
```

**关键设计**：先展示 AST 骨架（零延迟），LLM 结果到达后无缝替换。

---

## 一、触发入口

**文件**: `frontend/src/components/UploadBar.tsx` → `handleVisualize()`

```
1. addResponse({action: 'visualize', loading: true})   ← 创建响应卡片，显示 loading
2. analyzeGraph()                                       ← 并行发出，拿到后立即展示骨架
     └→ updateResponseContent(id, JSON.stringify(skeleton))
3. await analyzeOverview()                              ← 等 LLM 返回语义图
     └→ updateResponseContent(id, JSON.stringify(overviewData))  ← 替换骨架
4. setResponseDone(id)                                  ← 停止 loading
```

**展示侧**：`ResponseCard.tsx` 检测 `action === 'visualize'`，将 `content` JSON.parse 后传给 `<FlowChart>`。

## 二、后端 4 个 API

所有端点在 `backend/routers/analyze.py`，共享同一个缓存的 `CallGraph`。

### 2.1 `/api/analyze/graph` — 纯 AST 调用图

```
_get_or_build_graph()
  ├─ build_call_graph(project_files)     ← call_graph.py, 4 步流水线
  └─ detect_entry_points(graph)          ← entry_detector.py

_build_flow_data(graph)
  ├─ Layer 1: 模块级 (每个 .py = 节点, import = 边)
  └─ Layer 2: 函数级 (每个函数 = 节点, 调用 = 边, 按模块分组)
```

**返回结构**:
```json
{
  "modules": { "path": { "line_count": 50, "imports": [...] } },
  "definitions": { "file::func": { "name", "kind", "line_start", "line_end", ... } },
  "edges": [ { "caller", "callee_resolved", "file", "line" } ],
  "flow": {
    "module_level": { "nodes": [...], "edges": [...] },
    "function_level": { "file_path": { "nodes": [...], "edges": [...] } }
  }
}
```

**特点**：零 LLM 调用，毫秒响应。

---

### 2.2 `/api/analyze/overview` — LLM 语义流程图

```
graph_dict = graph.to_dict()
system_prompt, user_prompt = build_overview_prompt(graph_dict)
    └─ 只发送函数签名 + docstring + 调用关系（不发源码）

raw = call_qwen(system_prompt, user_prompt)    ← LLM 生成语义图
data = _parse_llm_json(raw)                    ← 去掉 markdown 格式，解析 JSON
_normalize_flow_data(data)                     ← id/source/target 强转 string，过滤无效边
_fill_line_numbers_from_ast(data, graph)       ← 关键：用 AST 精确行号覆盖 LLM 可能错误的行号
```

**LLM 输入**（精简后）:
```
项目调用图骨架：
- upload_file (routers/file.py:18-30) → validate_file, store_file
- review_code (routers/review.py:16-42) → get_related_files, build_review_prompt

每个函数的签名和 docstring：
- validate_file(filename, content): "验证上传文件"
...
```

**LLM 输出**: `{ nodes: [...], edges: [...] }` — 包含 start/process/decision/end 节点。

**行号修正** (`_fill_line_numbers_from_ast`):
- LLM 返回的 `symbol` 字段 → 在 `graph.definitions` 中查找 → 填入精确的 `lineStart`/`lineEnd`
- 三级匹配：qualified_name 精确匹配 → 同文件同名 → 全局同名

---

### 2.3 `/api/analyze/annotate` — LLM 语义标注

```
给 call graph 的每个函数生成中文 label + description
可选按 module 过滤，减少 token

成功 → { status: "ok", annotations: { "file::func": { label, description } } }
失败 → { status: "fallback", annotations: { 用函数名作为 label } }
```

---

### 2.4 `/api/analyze/detail` — 展开函数内部逻辑

```
输入: qualified_name (如 "backend/routers/file.py::upload_file")

1. 从 graph.definitions 找到函数定义
2. 提取该函数的源码 (仅这一个函数，不是全部项目)
3. 收集它调用的函数的签名
4. build_function_detail_prompt() → LLM
5. LLM 返回该函数内部的 start/process/decision/end 流程图
```

**与旧 API 的区别**：旧的 `/api/visualize/detail` 发送全量项目代码给 LLM，新的只发目标函数源码 + 被调函数签名。

## 三、AST 调用图构建

**文件**: `backend/services/call_graph.py` → `build_call_graph()`

### 4 步流水线

```
Step 1: 提取定义
  ast.parse(source) → 遍历 FunctionDef / AsyncFunctionDef / ClassDef
  → SymbolDef { qualified_name, name, kind, file, line_start, line_end,
                 decorators, docstring, params }

Step 2: 提取调用边
  遍历每个函数体内的 ast.Call 节点
  → CallEdge { caller, callee_name, file, line, call_type }
  call_type: "direct" (foo()) 或 "attribute" (self.foo())

Step 3: 解析调用目标
  结合 import 分析，将 callee_name 解析到项目内的具体函数
  import os → 排除（外部库）
  from backend.services.llm import call_qwen → 解析到 "backend/services/llm.py::call_qwen"
  → 设置 edge.callee_resolved

Step 4: 构建模块图
  每个文件 → ModuleNode { path, line_count, symbol_count, imports[] }
  imports = 该文件 import 了哪些项目内文件
```

### 核心数据结构

```python
CallGraph:
  definitions: dict[str, SymbolDef]   # "file::func" → 定义信息
  edges: list[CallEdge]               # 所有调用关系
  modules: dict[str, ModuleNode]      # 模块级依赖
  entry_points: list[str]             # 入口函数的 qualified_name
```

## 四、入口点检测

**文件**: `backend/services/entry_detector.py` → `detect_entry_points()`

4 条规则，优先级从高到低：

| 规则 | 匹配条件 | 示例 |
|------|----------|------|
| 路由装饰器 | `@app.get`、`@router.post` 等 | FastAPI 端点 |
| CLI 装饰器 | `@click.command`、`@app.command` | CLI 命令 |
| Main Guard | `if __name__ == "__main__":` 所在文件的顶层函数 | 脚本入口 |
| 孤儿函数 | 没有被任何其他函数调用的顶层函数（排除私有 `_func`） | 工具函数 |

检测结果写入 `SymbolDef.is_entry` 和 `graph.entry_points`。

## 五、前端渲染

### 5.1 FlowChart 组件

**文件**: `frontend/src/components/Diagrams/FlowChart.tsx`

```
FlowData (props)
    │
    ├─ computeLayout(data)        ← Dagre 分层布局 (rankdir: TB, 从上到下)
    ├─ buildNodes(data, selId)    ← 转换为 ReactFlow Node 格式
    ├─ buildEdges(data)           ← 转换为 ReactFlow Edge 格式
    │     └─ decision 的"否"分支从右侧出边 (sourceHandle: 'no')
    └─ ReactFlow 渲染
```

### 5.2 节点交互

```
单击节点 (handleNodeClick):
  ├─ 如果是已选中的可展开节点 → enterSubFlow() (进入子图)
  └─ 否则:
       ├─ 选中该节点 (高亮)
       └─ 跳转代码:
            ├─ 有入边 call_line → 跳到调用位置 (caller 文件中调用该函数的那一行)
            └─ 无入边 (入口节点) → 跳到函数定义处 (lineStart)
```

### 5.3 子图展开 (enterSubFlow)

```
双击可展开节点:
  ├─ 缓存命中 → 直接切换 currentData
  └─ 缓存未命中:
       ├─ 调用 visualizeDetail() ← 注意：目前仍用旧 API
       ├─ 结果写入 cacheRef
       └─ 切换 currentData + 更新 navPath (面包屑)
```

### 5.4 节点类型与样式

**文件**: `frontend/src/components/Diagrams/CustomNode.tsx`

| 类型 | 形状 | 颜色 | Handle |
|------|------|------|--------|
| start/end | 胶囊 (圆角矩形) | 灰色 | 单向 |
| process | 矩形 | 蓝色 (可展开为紫色) | 上+下 |
| decision | 菱形 (45°旋转) | 蓝色 | 上+"是"(下)+"否"(右) |

### 5.5 侧边栏树形导航

**文件**: `frontend/src/components/Diagrams/FlowTreeNav.tsx`

- 递归树形结构，只显示 `process` 和 `decision` 节点
- 单击聚焦，双击展开
- 图标：`▼` 已展开 / `◇` 可展开 / `⏳` 加载中 / `·` 普通

## 六、Prompt 设计

**文件**: `backend/services/prompts/annotate.py`

| Prompt 构建函数 | 用于 | 输入 | Token 量 |
|-----------------|------|------|----------|
| `build_overview_prompt()` | `/api/analyze/overview` | 函数签名 + 调用关系 | 少 |
| `build_annotate_prompt()` | `/api/analyze/annotate` | 函数签名 + docstring | 少 |
| `build_function_detail_prompt()` | `/api/analyze/detail` | 单个函数源码 + 被调函数签名 | 中 |

旧 Prompt（`prompts/visualize.py`）发送全量源码，Token 消耗大。

## 七、数据流总结

```
项目源码
  │
  ▼
ast.parse() ──────────────────────────────────────────────┐
  │                                                        │
  ├─ SymbolDef[] (函数/类定义, 精确行号)                      │
  ├─ CallEdge[] (调用关系, 调用行号)                          │
  ├─ ModuleNode[] (模块依赖)                                │
  └─ entry_points[] (入口检测)                              │
  │                                                        │
  ▼                                                        │
CallGraph ─────────────────────────────────────────────┐   │
  │                                                     │   │
  ├─→ _build_flow_data() ──→ module_level FlowData     │   │
  │                     └──→ function_level FlowData    │   │
  │                                                     │   │
  ├─→ build_overview_prompt() ─→ LLM ─→ 语义 FlowData  │   │
  │                                         │           │   │
  │                    _fill_line_numbers ◄──┘           │   │
  │                    (AST 行号覆盖 LLM 行号)            │   │
  │                                                     │   │
  └─→ build_function_detail_prompt() ─→ LLM ─→ 子图     │   │
                                                        │   │
                                                        ▼   ▼
                                                 前端 FlowChart
                                                 Dagre 布局
                                                 ReactFlow 渲染
```
