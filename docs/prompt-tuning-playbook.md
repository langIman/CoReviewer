# Prompt 调优手册

本文档总结**模块页（module page）prompt 调优**这轮工作的方法论与可复用经验，供后续打磨核心架构页（chapter page）与专题深入页（topic page）时直接对照使用。

**本轮成果定位**：iter4 的模块页 prompt（[wiki_prompts.py](../backend/services/llm/prompts/wiki_prompts.py) `build_module_page_prompt`）= **single-shot LLM 调用框架下的合理顶点**——再调收益已经很小，剩余差距属于模型能力天花板（需要架构升级到 ReAct 才能突破）。

---

## 一、两条独立的优化轴

任何文档生成质量都由两条轴决定：

```
质量 = f(信息源, 输出 prompt)
```

**这两条轴必须分开看待**：
- 信息源决定 LLM "能看到什么"——再好的 prompt 也救不了缺失的信息
- 输出 prompt 决定 LLM "怎么组织"——再丰富的信息也需要好的写作引导

调优时先**确认信息源完整且真实进了 prompt**，再调输出 prompt。

### 信息源管线的隐患

模块页这轮发现 `target_length_hint` 是**死参数**——签名声明了，模板里没占位符，传了等于没传。这暴露了一个普遍隐患：**输入参数和模板占位符可能脱钩**。

**SOP**：每加一条新的输入信息，运行 `dump_module_0_prompt.py` 之类的脚本 dump 出真实 prompt，搜一下这个值是否真的进入了文本。

### 信息源的复用情况

| 信息源 | 模块页 | 核心架构页 | 专题页 | 备注 |
|---|---|---|---|---|
| 项目摘要 | ❌ | ✅ | ✅ | 给业务上下文 |
| 模块清单 + 描述 | ✅（自身） | ✅ | ✅ | 用于跨链接 |
| 完整源码 | ✅ | ❌ | ⚠️ 应聚焦 | chapter 用模块摘要替代 |
| 跨模块调用关系 | ✅（已渲染） | ⚠️ 当前缺 | ❌ | **chapter 应当加上** |
| README 片段 | ✅ | ✅ | ⚠️ | 业务背景 |
| 允许的页面 id | ✅ | ✅ | ✅ | 控制跳转合法性 |
| 文件路径列表 | ✅ | ❌ | ⚠️ | 校验用 |
| **演进/历史线索** | ✅ 注释代码 | ⚠️ 应加 git log 摘要？ | ⚠️ 该专题的设计演进 | **高价值角度** |

**对 chapter / topic 页的信息源建议**：
- 核心架构页当前**没给跨模块调用频次**——可以加。模块页里 "→ module_4 (7 处调用)" 是 LLM 的有用线索
- 专题页是**最聚焦**的页面（针对某一具体设计点），输入应当**只给相关代码片段**而非全项目，避免注意力分散

## 二、输出 prompt 的六条核心技巧

按重要性排序。每条都从模块页 iter1-iter4 的真实迭代里提炼出来。

### 1. 视觉层次是最大的单一杠杆

不是抽象的"结构清晰"，而是**多种具体视觉手段并用**：

| 手段 | 用途 | 何时用 |
|---|---|---|
| H2 / H3 / H4 | 章节切分 | 永远。H4 用于"H3 内有 2+ 子话题"时 |
| 引用块（`> `） | 5 类侧栏：细节 / 失败模式 / 设计权衡 / 关键观察 / 小结 | 想从正文剥离的补充材料 |
| 表格 | 对比 / 配置 / 失败语义 | 平铺会冗长的对比关系 |
| ASCII 框图 | 内存布局 / 比特结构 / 三层架构示意 | mermaid 不擅长的"静态结构图" |
| mermaid | 时序 / 流程 / 状态机 | 动态过程 |

**各司其职、不互相替代**——一份高质量页面通常 mermaid + 1-2 张表 + 偶尔 ASCII 框图 + 3-8 处引用块。

#### 章节密度的硬约束

LLM 默认倾向于"塞"——把多个独立话题塞进 1 个 H2 用 H3 凑数。必须给硬下限：

```
≤6 文件/模块 → 至少 3 个 H2
7-12 → 至少 4 个 H2
≥13 → 至少 5 个 H2
```

迁移到其他页面时，按**"概念密度"**调整门槛（核心架构页可能按 H2 数对应章节主题数；专题页通常 H2 较少但 H3/H4 切分更细）。

### 2. 「概览—细节」节奏

每个 H2 进入子节前**强制**一段 2-4 句引导段，告诉读者本节要看什么、为什么这么切、和上一节的关系。

```
## 3. 异步下单：Stream + 单线程消费

资格判定通过后，订单落库**完全异步**——同步路径返回的瞬间，DB 还没写。
Stream + Worker 这套组合是 v4 的 BlockingQueue 升级而来，每一处选择都对应 v4 踩过的坑。

### 3.1 为什么不是 BlockingQueue
...
```

**这是技术文档"可读性"的关键标志**。从 [Claude Code 上下文工程文档](https://diwang.info/how-claude-code-works/#/docs/03-context-engineering)学到的——它每个 H2 都先给全景再下钻。

迁移时：
- 核心架构页因为是叙事流，引导段比模块页**更重要**
- 专题页可以稍轻，因为整篇就一个主题

### 3. 自查清单是硬约束的最佳载体

**关键发现**：LLM 经常在数 ref / 数表格 / 数 H2 上偷懒。光说「至少 N 个」无效——必须在 prompt 末尾给一个**「写完后必须执行的自查清单」**。

```
**写完 detail_md 后必须执行的自查清单**（顺序检查）：
1. 数 ref：从 ref_1 到 ref_N 数一遍，N 必须 ≥ {min_code_refs}。如果不够，回去补——
   常见漏算：辅助工具类、被引用的常量、状态字段
2. 数引用块：全文 `> ` 引用块应当 3-8 处。如果是 0-2 处，回去把至少 2 个
   "普通段落里的设计权衡 / 失败模式警示"提取成引用块
3. 数表格：除速览区外应当有 1-3 张表
4. 数 H2：按章节密度下限自查
```

iter4 把 code_refs 从 7 拉到 9 就是这个清单的直接效果。

**这是这轮最实用的技巧。** 任何"硬指标"都应该用这种方式收尾。

迁移时：每种页面定一组该自查的"硬指标"。例：
- 核心架构页：mermaid 至少 1 张 / 跨模块跳转至少 N 个 / 章节字数下限
- 专题页：替代方案分析数 ≥ 2 / 失败模式数 ≥ 2 / 「为什么」型洞察数 ≥ 5

### 4. 示例 > 抽象规则

**LLMs 看示例比看规则更听话。** 加新规则时配 1-2 个具体示例，效果远好于纯描述。

#### 三种示例形式

**形式 1：✗/✓ 反例对**

```
**禁止"是什么"式总结**：
- ✗ "使用 Lua 脚本保证原子性"
- ✓ "Lua 替代 MULTI/EXEC 的根因是事务不能中途分支——而秒杀必须有
   if 库存>0 then 扣减 else 失败 这种条件分支"
```

iter4 直接照抄了这个句式作为设计洞察第 1 条。

**形式 2：mini-template**

```
示例结构（节选自真实模块页）：
  ## 1. 双拦截器协作模型
  ### 1.1 RefreshTokenInterceptor：上下文恢复
  ### 1.2 LoginInterceptor：强制鉴权
```

LLM 会模仿这种结构格式。

**形式 3：触发条件示例**

```
H4 触发条件：
- 一个 H3 内有 2+ 个明显独立的子话题
  （如「替代方案对比」下「为什么不用 X」「为什么不用 Y」）
- 一个 H3 描述的失败模式有"主路径"和"补偿路径"两条线
```

让 LLM 知道**什么时候**用某个手段，比强制要求更有效。

### 5. 反退化清单（禁止式约束）

**每种页面有自己的典型失败模式**。必须先观察 baseline 输出，再针对性加禁止条款。

模块页这轮加的禁止条款（都是真实踩过的坑）：

- 禁止「全文都很重要」式的 reading_guide
- 禁止从模块编号 / 文件名 / 变量名脑补业务含义
- 禁止把入门级做法（加盐 MD5 / Redis 缓存）包装成亮点
- 禁止 emoji 起头标题
- 禁止纯粹复述源码而不分析
- 禁止重复速览区已有事实

**对其他页面的提醒**：
- 不要从模块页直接复制禁止条款
- 先跑 1-2 次 baseline，看 LLM 在这种页面上**怎么失败**，再针对性加
- 否则 prompt 会越来越臃肿，LLM 注意力越来越散

### 6. 设计洞察：「为什么」vs「是什么」

**LLM 默认倾向写「是什么」，必须强约束逼出「为什么」。**

模块页这轮的具体做法：
- 数量下限：5-8 条
- 用 ✗/✓ 反例对展示差别
- 强调"设计洞察是其他正文章节的元结论"

**这是模块页 / 核心架构页 / 专题页质量的元约束**——但三种页面的应用比例不同：

| 页面类型 | 「为什么」型内容占比 |
|---|---|
| 模块页 | 末节集中（5-8 条洞察） + 散布在正文 |
| 核心架构页 | 散布在叙事流中 + 收尾点题 |
| 专题页 | **整篇都是「为什么」**——这是它和其他页面的本质区别 |

## 三、调优方法论（SOP）

工作流：

```
1. 先建 Gold 参考（不要盲调）
   - 用真实输入数据手写一份"理论上能达到的最优"
   - Gold 是后续所有迭代的"对照标尺"
   - 没有 Gold 就只能凭感觉调，无法量化进展

2. 系统对比 baseline vs Gold
   - 列具体维度（H4 数 / 表格数 / 引用块数 / 内容深度等）
   - 不写抽象的"层次感不够"——要落到可数的指标

3. 每轮只改 2-3 条
   - 改太多无法归因（这条改进是哪条 prompt 改动带来的？）
   - 改太少边际收益低
   - 当某条改动效果好，留下；效果差或矛盾，回滚

4. 用真实测试脚本跑实测
   - test/scripts/preview_module_page.py --refresh-page
   - 一轮 40-180s（看 thinking on/off）
   - 不靠"我觉得 LLM 应该会..."——真跑

5. 知道何时停
   - 当一轮迭代的边际收益明显下降
   - 当剩下的差距是"模型能力"而非"prompt 表达"
   - prompt 越长 LLM 注意力越散，过度优化反伤
```

### 测试基础设施（已就绪）

模块页的迭代测试链路已经建好：

- [`test/scripts/preview_module_page.py`](../test/scripts/preview_module_page.py) — 跑单模块生成
- [`test/scripts/dump_module_0_prompt.py`](../test/scripts/dump_module_0_prompt.py) — dump 真实 prompt 输入
- `test/.cache/split_TestProject.json` — split 缓存稳定，模块索引固定
- `test/.cache/module_page_llm_*.json` — 按 prompt+model 哈希缓存 LLM 响应，prompt 改动自动失效

**核心架构页 / 专题页需要建对应的预览脚本**——参照 `preview_module_page.py` 的模式：
- 装载文件 + 复用 AST/摘要缓存
- 调对应的 generate 函数
- 写到 `/tmp/iter*.md` 方便对比

## 四、对核心架构 + 专题深入的具体建议

### 第一步：先建 Gold（最重要）

为每种页面**手写 1 份 Gold**：

**核心架构页 Gold**：
- 选一个有真实跨模块叙事价值的章节（避开 boilerplate 章节如「数据访问层」）
- TestProject 项目里可以选「秒杀订单的端到端流程」这种横跨多模块的话题
- Gold 应当展示：mermaid 跨模块图 + 各模块在流程中的角色 + 关键交接点的设计取舍

**专题深入 Gold**：
- 选一个有"非显然设计"的主题（不是常规功能介绍）
- TestProject 里可以选「秒杀的多层防御策略」「Redis 在系统中的 SPOF 角色」之类
- Gold 整篇都是「为什么这样而不是别的」——犀利、聚焦、有立场

### 第二步：跑 baseline + 系统对比

跑当前 prompt 的 baseline，按下表对比：

| 指标 | Baseline | Gold | 差距 |
|---|---|---|---|
| H2 / H3 / H4 数 | | | |
| 引用块数 | | | |
| 表格数 | | | |
| mermaid 数 | | | |
| 跨模块跳转数 | | | |
| 「为什么」型分析占比 | | | |
| 替代方案对比次数 | | | |
| 内容覆盖（核心点） | | | |

### 第三步：迁移六条技巧

按优先级套用，根据各自页面特性调整：

| 技巧 | 模块页 | 核心架构页 | 专题页 |
|---|---|---|---|
| 视觉层次主力 | H4 + 表格 | mermaid + 引用块 | 反问 H3 + 对比表 |
| 概览节奏 | 每 H2 引导段 | 每章导语段（更长） | 通篇主题论证 |
| 设计洞察 | 5-8 条收尾 | 跨模块洞察收尾 | **整篇都是洞察** |
| 自查清单硬指标 | refs / 表格 / H2 数 | mermaid / 跨模块跳转 / 字数 | 替代方案数 / 失败模式数 |
| 演进轨迹 | 注释代码 | 项目级重要重构 | 该专题的设计演进 |
| 反退化清单 | 模块特有失败 | 跨模块特有失败 | 「水文」/「教科书化」 |

### 第四步：迭代，每轮 2-3 条改动

按模块页这轮的节奏：iter1（结构层次）→ iter2（章节密度）→ iter3（失败假设）→ iter4（自查清单）。

## 五、不该再追的天花板

诚实标记一下：**有些质量差距是模型能力问题，不是 prompt 问题**。这轮模块页发现的 4 项 Gold 独有但 prompt 拉不到的观察：

- **跨段落连接观察**（如 `@Transactional` + 分布式锁 ordering bug）
- **隐式假设挖掘**（如 `Collections.emptyList()` 暗示 Redis Cluster 不兼容）
- **写后元认知**（如「取件码 + 快递柜」的心智模型）
- **冷门技术栈的深度**（模型预训练分布决定）

这些都需要 ReAct 架构（多步观察-推理）才能突破，**不是 prompt 调优能解决的**。

承认天花板比硬撑 prompt 健康得多——避免在已经达标的部分过度优化反而损害平衡。

## 六、已知架构层选项（未来路线图）

如果某天「single-shot 不够用」成为产品定位上的真实痛点，按成本递增有三档：

| 方案 | 描述 | 成本 | 期望 lift |
|---|---|---|---|
| A. 两步小管线 | Step1 侦察输出观察点，Step2 写作 | 2x LLM 调用 / ~80s | 抓回 50-75% Gold 独有观察 |
| B. 分章节深挖 | 大纲 → 各章独立深挖 → 合成 | N+2 次 / 4-5min | 抓回 75% + 章节深度均匀 |
| C. Tool-using agent | 给 LLM `read_file_chunk` 工具自主探索 | 5-15 次 / 5-10min | 接近 Gold 但不可控 |

项目已有 agent 基础设施（`backend/services/agent/`），方案 C 不算从 0 写。

**当前判断**：方案 A 性价比最高，但应该先验证 single-shot iter4 是否真的不够用——可能 80% Gold 质量在生产中已够用。
